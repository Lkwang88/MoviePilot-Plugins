# GD网盘Strm助手 (GDStrmHelper)

为 **rclone 挂载的谷歌网盘** 生成 `.strm` 文件的 MoviePilot V2 插件。基于 `CloudStrmCompanion` 重构，剥离 115/alist/cd2 等无关逻辑，专注 rclone GD 场景，重点优化了**异步启动、增量扫描、并发处理、删除安全**。

---

## 核心特性

- **异步启动，不阻塞 MP**：插件初始化秒返回。监控与首次全量扫描在后台线程中延迟执行，重挂谷歌网盘后 MoviePilot 不必等待全盘索引即可正常启动。
- **SQLite 状态记忆 + 增量扫描**：用 SQLite 记录每个文件的 `size + mtime + 产物路径`。**媒体与非媒体（nfo/jpg 等元数据）全部纳入记忆**，第二次以后的扫描只处理新增/变更文件，已处理的直接跳过，大库二次扫描耗时大幅下降，中断后天然断点续扫。即使 SQLite 丢失，只要本地产物已存在也会跳过并补库——**本地已有的任何文件都不重复处理/下载**。
- **生产者-消费者并发**：扫描/监控只负责把变更文件塞进队列，由工作线程池并行生成 strm。**没有全局锁**，单个慢 I/O 不会卡住整体。
- **实时监控双模式**：默认**轮询模式（PollingObserver）**，对 rclone 挂载的**云端新增敏感**（不依赖 inotify 事件，靠定期 stat 对比目录，rclone 缓存刷新后下一轮就能捕获，配合 `--poll-interval` 接近无延迟）；也可切换 **inotify 模式**（省 CPU、可休眠，仅适合本地磁盘/只有本机写入的场景）。轮询间隔可配。
- **定时增量扫描兜底**：作为监控漏事件的保险。默认每 30 分钟一次，配合 SQLite 记忆几乎不耗资源。
- **Emby 聚合刷新**：一段安静期内新增的 strm 合并成一次 `Library/Media/Updated` 通知，避免 API 轰炸。支持 MP 侧与 Emby 侧的 strm 路径映射。
- **通知聚合防轰炸**：新增媒体攒一批，安静期后发一条汇总通知（剧集自动合并季集）。
- **删除同步三重保险**（生产级媒体库安全设计，删除是破坏性操作）：
  1. 删除前对每个挂载目录做**多次存活检查**（存在 + 非空），任一次失败则整轮放弃，杜绝 rclone 掉线导致误删。
  2. **数量熔断**：待删除数达到阈值时整轮放弃（0 删除）并告警，防止异常批量误删。
  3. **目录安全边界**：只删除 DB 记录的 `.strm` 文件本身；向上清理空目录时用 `commonpath` 严格判定，**绝不删除 STRM 生成根目录本身（如 `/media/LK01`，即使为空）**，更不会触及 `/media` 或任何上级/无关目录；strm 若不在其所属生成目录内（配置变更等）则直接跳过不删。
- **原子写入 + 幂等**：strm 先写临时文件再 rename；内容无变化不重写、不触发刷新。
- **scandir 遍历**：比 `os.walk` 更省 stat 系统调用，对 rclone 挂载更友好。

## 配置说明

### 目录配置（每行一个网盘）

```
监控目录#STRM生成目录#Emby播放路径#[Emby STRM目录(可选)]
```

| 段 | 含义 | 示例 |
|---|---|---|
| 监控目录 | MP 里 rclone 挂载的谷歌网盘路径 | `/opt/fufu/gd1` |
| STRM生成目录 | MP 里存放 `.strm` 的目录 | `/media/strm/gd1` |
| Emby播放路径 | 写进 strm 内容里的路径，即 **Emby 容器看到的网盘挂载路径** | `/mnt/gd1` |
| Emby STRM目录 | （可选）Emby 容器里看到 `.strm` 文件的路径，用于刷新通知。留空 = 与"STRM生成目录"相同 | `/strm/gd1` |

示例：

```
/opt/fufu/gd1#/media/strm/gd1#/mnt/gd1
/opt/fufu/gd2#/media/strm/gd2#/mnt/gd2#/strm/gd2
```

> strm 内容 = 把"监控目录"前缀替换为"Emby播放路径"。例如源文件 `/opt/fufu/gd1/电影/A.mkv` → strm 内容 `/mnt/gd1/电影/A.mkv`。

### 开关

| 开关 | 说明 |
|---|---|
| 启用插件 | 总开关 |
| 实时监控 | 启动 watchdog 监听文件变化（默认轮询模式，见下方"实时监控 vs 增量扫描"） |
| 发送通知 | 新增媒体聚合后发汇总通知 |
| 刷新Emby | 聚合刷新 Emby 媒体库 |
| 覆盖已存在STRM | 强制重写已存在的 strm（默认关，幂等跳过） |
| 复制非媒体文件 | 复制 nfo/jpg 等到 strm 目录 |
| 复制字幕 | 复制 srt/ass 等字幕 |
| 删除同步 | 云端删除的文件自动清理对应 strm（默认关，带双保险） |
| 立即全量扫描一次 | 手动触发一次全量 |
| 立即增量扫描一次 | 手动触发一次增量 |
| 立即清理孤儿STRM | 手动触发一次删除同步 |

### 数值参数

| 参数 | 默认 | 说明 |
|---|---|---|
| 实时监控模式 | 轮询 | 轮询(rclone 推荐，对云端新增敏感) / inotify(省 CPU，仅本地磁盘) |
| 轮询间隔(秒) | 10 | 仅轮询模式生效，越小越及时、开销越大 |
| 启动延迟(秒) | 60 | 等 MP 完全启动后再执行监控与全量扫描 |
| 工作线程数 | 4 | 并发生成 strm 的线程数 |
| 通知聚合延迟(秒) | 10 | 剧集攒够这段静默期后发通知 |
| Emby刷新安静期(秒) | 30 | 攒够这段安静期后合并刷新 Emby |
| 增量扫描周期(cron) | `*/30 * * * *` | 定时增量扫描的 cron 表达式（默认 30 分钟，作为轮询漏事件的兜底） |
| 删除前挂载检查次数 | 3 | 删除同步前连续检查挂载存活的次数 |
| 单轮最大删除数(熔断) | 10 | 待删除数达到此值则整轮放弃删除并告警 |

## rclone 挂载建议

配合以下参数，增量扫描又快又新：

```
--dir-cache-time 36h      # 目录缓存久 = 扫描读缓存飞快
--poll-interval 1m        # 轮询刷新缓存 = 云端新增约1分钟内反映到挂载
--vfs-cache-mode full
```

> 说明：rclone 轮询刷新缓存时不会向内核发 inotify 事件。**轮询模式的 watchdog（默认）不依赖 inotify 事件，靠定时 stat 对比目录**，因此只要 rclone 缓存刷新让文件出现在挂载里，下一轮轮询就能捕获——这就是它对"别机传到 GD 的云端新增"也敏感的原因，体感接近无延迟。inotify 模式则只能捕获经过本机 FUSE 层的写入，对云端新增是瞎的（这类只能靠增量扫描兜底）。

## 版本历史

- **v1.6.0**：数据库维护与自愈。
  1. **启动自愈**：插件启动时对数据库做 `quick_check` 快检，若损坏则自动备份坏库（`.corrupt.时间戳`）保留现场并重建，避免带病运行。
  2. **手动维护开关**（配置页）：`检查数据库完整性`（quick_check + integrity_check，结果发通知）、`重建数据库`（清空重建后自动全量补库，凭本地已有产物跳过、不会重复生成 strm）、`数据库瘦身`（VACUUM 回收碎片 + 截断 WAL）。
  3. **删除记录立即落盘**：删除同步后强制 flush，避免崩溃窗口期删除记录丢失。
  4. 数据库仅为加速缓存：可随时安全删除 `gdstrm.db*`，重启全量会自动重建且不重复生成已有 strm。
- **v1.5.0**：大规模性能与可靠性优化（面向 20-40 万文件/盘）。
  1. **数据库连接复用**：从"每文件一次 connect"改为读写分离——全局唯一写连接 + 各线程持久只读连接，消灭数十万次建连接开销。
  2. **批量提交**：写入累计 500 条或 3 秒才 commit 一次，配 `synchronous=NORMAL`，写性能提升一个数量级（可靠性换性能：崩溃时未提交记录下次扫描自动重做，不会出错）。
  3. **读己之写缓存**：批量提交延迟期间，用内存缓存保证刚处理的文件不会被重复处理；提交后清空，缓存恒定只有一个批次大小，内存占用极小。
  4. **只读连接自动提交模式**（`isolation_level=None`）：修复 WAL 只读连接钉住旧快照、读不到新数据的问题。
  5. **扩展名预解析为 frozenset**：热路径不再反复 split 字符串。
  6. **有界队列 + 背压**：队列上限 1 万，扫描入队满则阻塞等待消费，限制内存峰值。
  7. **删除同步流式处理**：`fetchmany` 分批读 DB，不再一次性把几十万条读进内存。
  8. **WAL 检查点**：扫描后 `wal_checkpoint(TRUNCATE)` 回收 wal 文件，防止无限增长。
  9. **连接生命周期管理**：worker/扫描线程退出关只读连接，stop_service 关写连接，防止插件重载时连接泄漏。
- **v1.4.0**：删除同步安全加固（生产级）。空目录清理引入严格边界：用 `commonpath` 判定父子关系（杜绝 `/media/LK01` 误匹配 `/media/LK011`），**绝不删除 STRM 生成根目录本身及其外部任何目录**，只从 strm 所在目录逐层向上清理空目录、撞到生成根目录即停；strm 不在其所属生成目录内则跳过不删。新增 22 项删除边界专项测试。
- **v1.3.0**：
  1. 新增「保存即校验预览」：保存配置后立刻按每个盘算出路径映射（源文件→STRM 文件→STRM 内容→Emby 侧路径），输出到日志并推送系统消息（右上角通知），**在生成任何 strm 之前**即可核对映射是否正确。
  2. 新增配置合理性检查：监控目录不存在/为空、STRM 目录是监控子目录、Emby 播放路径未映射、开启 Emby 刷新但未选服务器等，均会给出提醒。
  3. 统计页「STRM 路径预览」保证无真实样例时也用占位示例展示，保存后随时可复查。
- **v1.2.0**：
  1. 实时监控新增「轮询 / inotify」模式选择，**默认轮询**（PollingObserver）——对 rclone 挂载的云端新增敏感，找回原插件近乎无延迟的体验；轮询间隔可配。
  2. 增量扫描默认周期改为 30 分钟（配合轮询监控，仅作漏事件兜底，更省资源）。
  3. 统计页新增「STRM 路径预览」：按当前配置展示每个盘的源文件→STRM 文件→STRM 内容→Emby 侧路径映射，扫描后自动采用真实样例。
- **v1.1.0**：
  1. 修复只启用插件但不开实时监控时，首次全量扫描不执行的问题（全量与监控解耦）。
  2. 非媒体文件（nfo/jpg 等元数据）纳入 SQLite 记忆，二次全量不再重复处理，只要本地已有就跳过。
  3. 增量判断增强：即使 DB 丢失，本地产物已存在也跳过并补库。
  4. Emby 刷新分批发送（每 100 个），防止全量时单次 payload 过大打挂 Emby。
- **v1.0.0**：首发。基于 CloudStrmCompanion 重构的 rclone 谷歌网盘专用版。异步启动、生产者-消费者线程池、SQLite 增量状态、scandir 遍历、inotify 优先监控、strm 原子写入、Emby 聚合刷新、通知聚合、删除同步（挂载多次存活检查 + 数量熔断）。
